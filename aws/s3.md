# S3 (Simple Storage Service)

object storage - you store files ("objects") in "buckets"
NOT a file system, NOT a database. think of it as infinite hard drive in the cloud

## core concepts

- bucket - top level container. name must be globally unique across ALL of aws
- object - the actual file + metadata. max 5TB per object
- key - the object's full path/name inside the bucket (e.g. users/123/avatar.png)
- there are no real "folders" - the slashes in keys just fake a folder structure
- region - bucket lives in one region. pick one close to your users

## why people use it

- store user uploads (images, videos, pdfs, csv exports)
- host static websites (html/css/js straight from a bucket)
- backups and logs
- data lake - dump raw data, query later with athena
- serve assets through a CDN (CloudFront in front of S3)

## the basics (CLI)

```
aws s3 mb s3://my-bucket-name          # make bucket
aws s3 cp file.txt s3://my-bucket/      # upload
aws s3 cp s3://my-bucket/file.txt .     # download
aws s3 ls s3://my-bucket/               # list
aws s3 sync ./folder s3://my-bucket/    # sync a whole folder
aws s3 rm s3://my-bucket/file.txt       # delete
```

## in code (python / boto3)

```python
import boto3
s3 = boto3.client("s3")

s3.upload_file("local.png", "my-bucket", "uploads/local.png")
s3.download_file("my-bucket", "uploads/local.png", "out.png")
```

## presigned URLs (IMPORTANT for production)

instead of routing big files through your own server, you hand the client a
temporary signed URL and they upload/download DIRECTLY to S3.

- saves your backend bandwidth + memory
- URL expires after X seconds
- this is how most "upload a profile pic" flows actually work

```python
url = s3.generate_presigned_url(
    "put_object",
    Params={"Bucket": "my-bucket", "Key": "uploads/abc.png"},
    ExpiresIn=3600,
)
# frontend then does a PUT request to `url`
```

## security stuff (don't skip)

- buckets are PRIVATE by default. keep it that way unless it's a public website
- "Block Public Access" = on. turning it off has leaked tons of company data
- use IAM roles/policies to control who can read/write
- enable encryption at rest (SSE-S3 or SSE-KMS) - usually on by default now
- bucket policy = rules on the bucket. IAM policy = rules on the user/role

## storage classes (cost optimization)

- Standard - hot data, accessed often
- Standard-IA (infrequent access) - cheaper to store, costs to retrieve
- Glacier / Glacier Deep Archive - cold backups, retrieval takes mins-hours, dirt cheap
- Intelligent-Tiering - aws auto-moves objects between tiers based on usage

## useful features

- versioning - keeps every version of an object, undo deletes/overwrites
- lifecycle rules - "move to Glacier after 30 days, delete after 1 year" automatically
- event notifications - fire a Lambda / SQS / SNS when an object is uploaded
  (e.g. user uploads image -> Lambda generates a thumbnail)

## gotchas

- bucket names are global - "my-bucket" is probably taken, namespace yours
- you pay for: storage + requests + data transfer OUT (egress). egress is the sneaky one
- eventual consistency used to be a thing; reads are now strongly consistent
- deleting a bucket requires it to be empty first
